### Critical findings (HIGH priority per Figma)

1. **LlmTabGroup — focus ring contrast 1.2 : 1.** Failing WCAG 2.4.7
   outright. The focus ring is essentially invisible against the tab
   background. Action: increase ring contrast to ≥ 3:1 in the Figma
   focus variant, then re-check the rendered Storybook story.
2. **LlmSelect — no focus variant in Figma.** A `state=focus` row is
   missing entirely. Likely the implementation has it (CDK Overlay
   default styles) but the design source of truth doesn't enforce it.
3. **LlmCombobox — no focus variant in Figma.** Same as Select.
4. **LlmButton — danger + active states differ only by colour.** WCAG
   1.4.1. Add an icon, border, or text indicator to either variant.
5. **LlmCheckbox — every variant 18 px tall.** WCAG 2.5.8 minimum is
   24×24. Same problem on **LlmRadio** (20 px), **LlmRadioGroup**
   (18 px).

### Low-priority — annotations & protanopia

- **A11y notes missing** in description: `LlmPagination`, `LlmTooltip`,
  `LlmDialog`, `LlmAlert`. Half of their annotations score is unused.
- **Protanopia contrast loss** (1-in-100 men): `LlmCodeBlock` (2 pairs),
  `LlmTabGroup` (2 pairs), `LlmPagination` (6 pairs). All on the same
  pattern: `#64748B` text → `#6D748C` simulated → 4.4 : 1 (just below
  4.5). Lift the colour to `#4f5f7c` or darker, mirroring the docs
  text-muted bump from commit `42356cc`.

## 4. Spec / implementation static audit (from prior Explore agent)

**Already correct — no changes needed:** LlmDialog, LlmInput,
LlmCheckbox, LlmRadio, LlmRadioGroup, LlmAlert, LlmToast, LlmTooltip,
LlmToggle.

**Three concrete one-prop fixes:**

1. **`LlmProgress`** — add `label?: string` to `LlmProgressSpec`,
   render as `aria-label`. Fixes the 3 `aria-progressbar-name`
   violations on /patterns.
2. **`LlmButton`** — TypeScript discriminated union: `children` XOR
   `'aria-label'`. Compile-time fail catches icon-only without name.
   Fixes the 2 `button-name` violations.
3. **`LlmAccordionItem`** — add
   `headingLevel?: 1 | 2 | 3 | 4 | 5 | 6` (default 3). React + Vue
   render `<h{headingLevel}>` dynamically; Angular uses
   `[attr.role]="'heading'"` + `[attr.aria-level]`. Fixes the 4
   `heading-order` violations.

**Open API design question:** `LlmCard` will get an opt-in
`role?: 'article' | 'region' | 'section'` prop (decided in the plan
discussion — default stays `<div>`). No spec change yet, but the
implementation needs to wire it once we touch the card.
